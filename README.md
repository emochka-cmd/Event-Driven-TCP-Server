Event-Driven TCP Server.
Это мой проект, на основе которого я хочу научиться взаимодействию с сетью, на практике понять и развить:
- как работает tcp;
- многопоточность и событийное дерево;
- навыки системного програмирования;
- развить навык описания архитектуры приложений;
- улучшу навыки управление памятью 

Пока архитектура планируется вот такая:
          Clients
             │
             ▼
      ┌──────────────┐
      │ TCP Listener │   accept()
      └──────┬───────┘
             ▼
   ┌───────────────────┐
   │ Event Loop (epoll)│  ← один поток
   └──────┬────────────┘
          │ events (read/write)
          ▼
   ┌───────────────────┐
   │ Connection Object │
   └──────┬────────────┘
          │ task
          ▼
   ┌───────────────────┐
   │   Thread Pool     │  ← N worker threads
   └───────────────────┘

Основные компоненты:
1. прослушиватель - проверяет наличие новых событий
2. отправитель задачи в pool
3. Connection (соединение) 
4. дерево с pool

Как это будет работать:
1. подключается клиент
2. ему присваивается fd
3. fd добавляеться в epoll
4. выполняется запрос пользователя
5. пользователю отдается ответ
6. соединение либо отключаеться, либо остается закрытым


Первый шаг - прослушка tcp и вопрос, как ее реализовать:
- создать сокет(т.е. виртуальный интерфейс, который позволяет обмениваться данными, использую ip). 
Возможные протоколы для реализации:
       1. TCP - протокол который гарантирует цельную доставку данных получателю.

       Сначала одна сторона отправляет запрос на соединение, 
       вторая подтверждает его получение и готовность к общению, а затем первая 
       подтверждает получение этого подтверждения.
       Только после этого начинается обмен данными. 
       Если не все данные были получены, они отправляются повторно.

       Этот процесс гарантирует, что соединение установлено надежно, 
       прежде чем начнется передача данных.

       2. UDP - протокол быстрой передачи данных, который не гарантирует 
       целостность доставки этих данных.
       
       При протоколе UDP нет никакого запроса на соединение и 
       подтверждение этого соединения, клиент отправляет данные вразнабой.
       
       Этот протокол используют, когда скорость передачи важнее надёжности
       соединения и целостность доставки данных.
       
       Например: для потоковой передачи видео.
В контексте данного проекта я хочу использовать tcp.
