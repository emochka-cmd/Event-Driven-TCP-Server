Event-Driven TCP Server

Описание проекта:

Мой проект,в котором я на практике разбираю низкоуровневое сетевое программирование и принципы работы высконагруженных сетевых приложений. 
Проект создаеться как платформа для экспериментов и глубокого понимания работы сети и сервера. Что планирируется реализовать и что уже реализовано - в файле queue.md.

Цели проекта:
- разработать и отладить TCP сервер;
- изучить event-driven модель и работу с epoll;
- принципы многопоточности;
- развить навыки сетевого програмирования;
- улучшить навыки проектировки архитектуры сетевых приложений;
- улучшить навыки управления памятью и ресурсами;

Общая архитектура:
```
          Clients
             │
             ▼
      ┌──────────────┐
      │ TCP Listener │ - accept()
      └──────┬───────┘
             ▼
     ┌─────────────────┐
     │      Epoll      │  ← один поток
     └───────┬─────────┘
             │ events   (read / write)
             ▼
    ┌───────────────────┐
    │ Connection Object │
    └──────┬────────────┘
           │ task
           ▼
   ┌───────────────────┐
   │   Thread Pool     │  ← N worker threads
   └───────────────────┘
```

Основные компоненты и за что они отвечают:
1. TCP Listener:
       - создание серверного сокета;
       - перевод сервера в неблокирующий режим;
       - закрепление сокета;
       - прослушивание;

2. Accept:
       - приём новых клиентских соединений (accept);
       - перевод клиентских сокетов в неблокирующий режим;

3. Epoll:
       - отслеживание событий;
       - реакция на события;

4. Connection Object, связан с конкретным fd:
       - хранит состояние клиента;
       - управление вводом/выводом;
       - состояния для обработки данных;
       - формирует список задач;

5. Thread Pool:
       - выполняет запросы пользоваетелей;

Логика работы сервера:
1. Клиент подключается к серверу.
2. Сервер принимает соединение и присваивает клиенту файловый дескриптор (fd).
3. fd регистрируется в epoll.
4. Event loop получает событие о готовности сокета.
5. Данные клиента читаются и формируется задача.
6. Задача передаётся в пул потоков для обработки.
7. Результат обработки отправляется клиенту.
8. Соединение:
       - либо остаётся открытым для следующих запросов,
       - либо корректно закрывается и очищается.
